프로미스 등장의 배경: 콜백 헬, 에러 처리

프로미스가 등장하게 된 배경에는 비동기 함수의 단점인 콜백 헬, 에러 처리의 어려움 등이 있습니다.

일반적인 비동기 함수에서 처리 결과를 활용해야 할 때, 비동기 함수는 외부에 그 결과를 반환할 수 없고, 상위 스코프의 변수에 할당할 수도 없습니다. 또, 비동기 함수는 에러 처리가 어렵습니다. 에러는 호출자 방향(콜 스택의 아랫쪽)으로 전파되는데, 비동기 함수의 컨텍스트는 호출자와 다르기 때문에 try catch 문으로 에러를 캐치할 수 없습니다.

따라서 항상 사후 처리를 하기 위한 콜백 함수를 내부에 전달해야 합니다. 그렇기 때문에 여러 번의 처리가 반복 수행되어야 하는 경우, 콜백 함수 여러 개가 중첩되어 복잡도가 높아지는 콜백 헬이 발생하게 됩니다.

이렇듯 비동기 함수는 콜백 헬로 인한 복잡한 구조, 낮은 가독성, 에러 처리의 어려움이라는 단점이 있습니다. 이를 개선하기 위해 ES6에서는 프로미스가 도입됩니다.

ES6, 프로미스

ES6에서는 프로미스가 도입됩니다.

const promise = new Promise((resolve, reject) => {
	//  비동기처리 수행 및 결과에 따른 함수 호출...
}));

프로미스 객체는 콜백 함수를 인자로 받는 생성자 함수를 통해 만들 수 있습니다. 인자로 전달받은 콜백 함수 내부에서 비동기 처리를 수행하며 그 결과에 따라 resolve, reject 함수가 실행됩니다.

프로미스의 상태

프로미스의 상태는 생성 직후, 즉 비동기 처리가 아직 완료되지 않은 경우 pending입니다. 처리가 완료된 상태는 settled 라고 통칭하며, settled 상태는 처리 성공 유무에 따라 fulfilled, rejected 상태로 다시 분류할 수 있습니다. 처리가 성공한 경우는 resolve 함수를 호출해 프로미스를 fulfilled 상태로 변경합니다. 비동기 처리가 실패한 경우는 reject 함수를 호출, 프로미스를 rejected 상태로 변경합니다.

프로미스 체이닝

일반적으로 프로미스를 반환하는 함수를 통해 비동기 통신을 구현합니다.

const water = temp => {
  return new Promise((resolve, reject) => {
    if (temp < 20) return resolve()
    return reject(new Error("Too Hot!"))
  })
}

이 메서드가 프로미스를 반환하도록 구현하였으므로 프로미스의 후속 처리 메서드인 then, catch, finally 등을 사용할 수 있습니다. 이때, then, catch, finally 메서드는 모두 또 다시 프로미스 객체를 반환하므로 연속적 호출, 즉 프로미스 체이닝이 가능합니다.

const coldwater = water(2)
  .then(() => console.log("Cooooooooool!"))
  .catch(error => console.log(error))
  .finally(() => "still thirsty!")

then 메서드는 인수로 두 개의 콜백 함수를 받으며, 각각 프로미스의 상태가 fulfilled, reject 일 때 실행됩니다. 프로미스 체이닝을 통해 연속적인 호출을 하는 경우, then 메서드의 두 번째 인자에서 reject 처리를 하면 콜백 함수에서의 에러를 캐치하지 못하는 경우가 발생할 수 있습니다. 이처럼 프로미스 체이닝이 반복되는 경우 catch 메서드로 체이닝하면 콜백 함수의 에러와 프로미스의 에러 두 경우 모두를 캐치할 수 있습니다.

then, catch, finally 메서드는 항상 프로미스 객체를 반환하므로 만약 콜백 함수가 프로미스 객체를 반환하지 않아도, 암묵적으로 프로미스를 생성하여 반환합니다.

Promise 객체의 메서드

Promise.resolve, Promise.reject

  • 프로미스 객체는 값을 래핑해서 프로미스를 생성해야 하는 경우 resolve, reject 정적 메서드를 활용합니다.

Promise.all

  • 인수: 프로미스를 요소로 갖는 이터러블로, 인수로 전달받은 이터러블의 요소가 프로미스가 아닌 경우에는 Promise.resolve 메서드를 통해 프로미스로 래핑합니다.
  • 반환: 모든 요소 프로미스들이 fulfilled 이면 처리 결과를 배열에 저장해서 새로운 프로미스로 반환합니다. 프로미스가 하나라도 rejected 상태가 되면 즉시 종료합니다.

Promise.race

  • 인수: 프로미스를 요소로 갖는 이터러블입니다.
  • 반환: 가장 먼저 fulfilled 상태가 된 프로미스의 처리 결과를 resolve 하는 새로운 프로미스를 반환합니다. 하나라도 rejected 되면 즉시 reject 하는 프로미스를 반환합니다.

Promise.allSettled

  • 인수: 프로미스를 요소로 갖는 이터러블입니다.
  • 반환: 모두 settled 상태가 되면 배열로 반환합니다. 배열의 요소는 프로미스 객체로, 프로미스의 상태가 fulfilled 라면 status, value 프로퍼티를 갖고, rejected 된 경우 status, reason 프로퍼티를 갖습니다.

기타

직렬화, 역직렬화

직렬화(serialize)는 객체를 JSON 형식의 문자열 형태로 변환하는 것이고 (JSON.stringify(obj)) 역직렬화(deserialize)는 JSON 형식의 문자열을 자바스크립트 객체로(JSON.parse(obj)) 만드는 것입니다.

마이크로태스크 큐

비동기 함수는 태스크 큐로 푸시되어, 콜 스택이 비었을 때 호출되게 됩니다. 마이크로태스크 큐는 태스크 큐와 유사하지만 보다 우선순위가 높은 큐입니다.

프로미스 후속 처리 메서드의 콜백 함수는 마이크로태스크 큐에 푸시됩니다. 콜 스택이 비면 마이크로태스크 큐에 있는 태스크가 콜 스택에 푸시되고, 마이크로태스크 큐가 비면 태스크 큐의 태스크가 콜 스택에 푸시됩니다.

fetch

fetch 함수는 프로미스를 반환합니다. 정확히는 HTTP 응답 객체(Reponse)를 래핑한 프로미스 객체를 반환합니다. fetch 함수는 resposonse.json() 메서드를 통해 역직렬화 단계를 후속 처리해줘야 하며, 에러가 발생해도 에러를 reject 하지 않고 response.ok 를 false로 설정한 Response 객체를 resolve 하므로, reject 처리를 위해 직접 response.ok 상태를 확인할 필요가 있습니다.

axios를 활용하면 catch 를 통해 모든 에러를 캐치할 수 있습니다.
